About

The uCracker is a very simple Java program that sniffs the network and displays by console the IDs and passwords of all of those that try to login to some sites in that same network.
It relies on the fact that the text that a common user puts in one of those sites, it is delivered to the network just as it's been written (it is not encoded in any way), so I just listen to the network (sniff) and display a password when some particular format is detected.



Get started

First of all, make sure you have installed Java 5 or heigher, then, once you have installed the application, run 

~$ java -jar uCracker.jar [-i interface] [-l loggingLevel] [--itba] [--fuckThemAll]

The -i parameter indicates what interface you want to sniff. We suggest you to specify one, because if not the program will search for one and there is no guarantee that interface is activated, or listening passwords.
If you don't now what interface to sniff, run the ipconfig (on Windows) or ifconfig (on any UNIX platform such as Mac or Linux) command on any shell and it will list all of your interfaces. Then choose one.

Some tips for choosing an interface:
  • If you are on a wireless network, such as WiFi, choose that interface that describes itself as wireless.
  • Choose that interface that is activated. (You will get it when you run the command)

The sites available to sniff have to be registered by adding very simple java classes to the project and recompileing (this will be MUCH BETTER in the future, this is the first release).

At last, but not least, the parameter -l indicates the logging level of the application and we suggest you not to indicate one and so the default value will be taken. The possible values for logging are

  • TRACE Logs a very detailed track of what is happening.
  • ERROR (default) Logs when errors happened.
  • OFF Don't log at all.

If everything went well, you will see something like this

~$ java -jar uCracker.jar -i en1
                    ...

Just keep this running on any terminal of yours and come back later.

How it works?

This is a very simple explication of how the uCracker works. If you want to know more about it, click below to see the detailed explication

Imagine a very big room, where there is one person on each corner of the room, and they want to send some messages to each other but the only way to do it it's by screaming the message. If they want to keep confidentiality into their messages, the won't scream their message just as it is, they will encrypt it in some way, because every other person in the same room will listen everything they said.

The same thing happens in computer networks. Each person it's a computer, and the room it's the network itself. And the analogy goes on, because a computer can listen to what other computer says, as long as they are on the same network, as you can see in this image.

IMAGE OF SNIFFING

So the problem here relies on how does a computer listen to another?. Here comes the concept of sniffing. To sniff is to listen to some determined interface on my computer(that's where the messages are being screamed) and not only to listen to the messages that come to me, but listen to them all. (Remember the analogy with the room, as you are on the same room you can listen to all that happens there). To set an interface to listen this way, is called listening in promiscuous mode.

So how to sniff? There are many ways. You could use wireshark or tcpdump, the first one has got a more "user-friendly" interface and the other one does it in command line. Or you could do an application that uses some library for sniffing, and analyze your packets as you like, like this application does.

uCracker uses a library called jpcap, which is a wrapper for Java of the library that actually sniffs, the libcap library.

So we have now an application that is able to listen to every messages on the room. The only thing to do it's to add some logic to recognize those messages that has been screamed just as they are, this is without any encryption, parse the ids and passwords, and display it to you.

Why did I do this?

My main goal here is to highlight two things:

  • CHANGE PERIODICALLY your passwords in your accounts.
  • DON'T REPEAT your passwords between your accounts.

There are many leaks and bugs in the secutiry systems of many applications, such as passing passwords as plain text, and you'll have a better result if you have some conciderations each both.
